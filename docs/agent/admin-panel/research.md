# Продуктовый ресерч: админ-панели для статичного сайта

Дата ресерча: 2026-06-10. Проверялись первоисточники (GitHub-релизы, документация
Sveltia / Pages CMS / Netlify / Cloudflare). Документ пополняется по ходу работ.

## Вердикт

1. **Кастомная vanilla-админка `/admin` + GitHub Contents API + деплой Netlify** — выбрано.
   Контент проекта доменно-специфичен (блоки кейсов, RU/EN пары, seeded-раскладка),
   готовые CMS дают типовые формы без живого превью реального сайта. Кастомная админка
   на том же стеке, что и сайт, реализует все UX-принципы ниже, а публикация = один
   коммит = автодеплой, откат = git-история.
2. **Sveltia CMS — план Б.** Один `admin/index.html` + `config.yml`, JSON-коллекции,
   лучший в классе i18n (RU/EN в одном файле, AI-перевод), drag-and-drop списков,
   WebP-оптимизация изображений. Миграция контента в JSON, которую мы делаем для
   кастомной админки, делает возможным и этот путь без переделок.
3. **Pages CMS — запасной hosted-вариант.** Самый дружелюбный интерфейс из лёгких
   git-CMS, вход по email-приглашению, но нет встроенного i18n — для двуязычного сайта
   это главный минус.

Не рекомендованы: Decap CMS (живой, но устаревший UX; Netlify Identity пережила
депрекацию-отмену, но для одного админа избыточна), TinaCMS и Keystatic (тянут
Node/React-сборку в проект без build-шага), Static CMS (архивирован 2024-09).

## 12 UX-принципов для нашей админки

Источники: Storyblok Visual Editor, Webflow Editor, Sanity Studio, Gutenberg
(включая критику), Notion-подобные редакторы, PatternFly inline edit.

1. **Редактирование в контексте превью.** Редактор видит страницу; клик по элементу
   открывает панель его полей. Минимум — живая превью-панель рядом с формой.
2. **Структурированные поля вместо свободного WYSIWYG.** Поля «Заголовок / Описание»
   не дают сломать вёрстку (антиурок Gutenberg).
3. **RU/EN бок о бок, а не табами.** Переключение табов между языками провоцирует
   рассинхрон переводов.
4. **Drag-handle, а не «вся карточка тянется».** Явная ручка, крупная зона захвата,
   плейсхолдер места вставки; дублирование стрелками вверх/вниз (доступность).
5. **Toggle с мгновенным визуальным откликом.** Выключенный кейс сразу затемняется и
   получает бейдж «скрыто» в списке.
6. **Замена медиа через drop-зону поверх текущего изображения**, с подсказкой целевых
   размеров и форматов. Никаких голых полей «путь к файлу».
7. **Автосохранение черновика + явная публикация.** Правки копятся локально, на сайт
   уходят только по кнопке «Опубликовать» (паттерн Webflow).
8. **Превью до публикации** и статус деплоя после неё прямо в админке.
9. **Откат = история версий из git.** Список публикаций с кнопкой «вернуть эту версию».
10. **Одна задача — один экран.** Список кейсов → карточка кейса → блоки; без вложенных
    меню; поиск сверху.
11. **Защита от потери работы и ошибок.** Предупреждение при уходе с несохранённым,
    валидация до публикации (пустой EN-заголовок, битый Vimeo ID), сообщения об ошибках
    на русском.
12. **Инлайн-правки для мелочей, форма — для крупного.** Заголовок и toggle — прямо в
    списке; полное редактирование — в карточке.

## Видео для статичного сайта

Лимиты: GitHub — блок пуша файлов больше 100 МБ, предупреждение от 50 МБ; Netlify Large
Media (Git LFS) депрекирован с 2023-09. Платные стримы: Cloudflare Stream (минимум
~$5/мес), Bunny Stream (~$1–3/мес для портфолио — самый дешёвый апгрейд при росте),
Mux (дороже, избыточен), Vimeo Free (1 ГБ на аккаунт, API-загрузка по заявке; Starter
$12/мес).

**Схема для нас** (сайт уже умеет локальные `.webm` и `vimeoId`):

1. Короткие лупы до ~20–30 МБ — файлами в репозиторий через админку (с предупреждением
   о размере).
2. Тяжёлые и длинные ролики — владелец загружает на Vimeo вручную, в админке вставляет
   ссылку/ID (автопарсинг URL → `vimeoId`). Автозагрузку через Vimeo API не строим.
3. При росте объёмов — Bunny Stream как самый дешёвый внешний хостинг.

## Drag-and-drop для vanilla JS

- **SortableJS** — стандарт де-факто: без зависимостей, MIT, тач-поддержка, стабилен.
  Для сортировки блоков кейса достаточно. Выбрано.
- Atlassian Pragmatic drag and drop — core-пакет framework-agnostic, активно развивается,
  но более низкоуровневый (визуальную обратную связь писать самим). Запасной вариант.
- dnd-kit — только React, не подходит. Dragula — почти не поддерживается.
